Problem Analysis

Problem 1: Snap Sound is Overloaded

The same snap sound + white flash is used for 7 different purposes: stop, blank, good, drop, nominal, touch, and silent pauses. After the first few Blank/Stop snaps, the conditioned response becomes snap → stop touching. But 12 snaps fire while touching that do NOT mean stop (silent vocalization pauses, "Good." reinforcements, "Touch." start). This creates compliance confusion.

Problem 2: Silent Snaps During Touch Are Ambiguous

16 silent snaps have no word displayed. 10 of these fire during active touching (mostly vocalization pauses: "say it inside", "the unit completes it"). The subject has no visual cue that this snap means "pause and vocalize" vs "stop touching". The automatic response is stop.

Problem 3: "Good." Snap at Line 249 During Touch

At line 249, @snap word:Good. fires while the subject is actively touching (they were told to touch at line 236 and told to keep touching through line 247). The snap + "Good." could be read as either "good, keep going" or trigger the conditioned snap→stop response.

Problem 4: No "Get Ready" Signal Before Edge/Touch Transitions

Transitions from no-touch to touch are abrupt: just text saying "Start touching." After extended no-touch periods (especially after drops), the subject needs a moment to physically prepare. There's no preparatory signal.

Recommendations

Recommendation 1: Add @pulseborder for Touch State

Add a pulsing colored border around the RSVP frame to indicate touch state. The border provides persistent ambient awareness of what you're supposed to be doing, without relying on memory of the last text instruction.

@pulseborder green hz:0.33        # Touching - slow green pulse
@pulseborder yellow hz:0.5         # Get ready (before "start touching")
@pulseborder red hz:0.75           # Edge / hold
@pulseborder off fade:1            # Not touching (default, no border)

This decouples the touch/no-touch signal from snaps entirely. The border is always visible, so even if the subject spaces out on a text instruction, the ambient color tells them what state they're in.

Recommendation 2: Differentiate Snap Types

Instead of one snap sound for everything, consider:

  • Keep snap.mp3 for compliance triggers (Blank, Stop) — these are the "stop now" moments
  • Use a softer/different sound for reinforcement (Good) and state triggers (Nominal)
  • Use NO sound for silent vocalization pauses — just show blank screen, no snap. Or use a gentle bell/tone.
  • This maps to the future @sfx command — scope it now, implement it to replace the overloaded snap
Recommendation 3: Add Prep Beats Before Touch Starts

Before each "start touching" instruction, add a 2-3 second yellow pulseborder to signal "get ready." This gives the subject time to physically prepare, especially after drops/blanks where they may be deep.

# Instead of:
Start touching yourself. Slow.

# Use:
@pulseborder yellow hz:0.5
Get ready.
@pulseborder green hz:0.33
Now. Touch. Slow.
Recommendation 4: Remove Silent Snaps During Touch

The 10 silent snaps during touching (vocalization pauses) should use a different mechanism. Options:

  • Just use a timed pause with no flash/sound (new @pause duration:2500 command)
  • Use a gentle bell (@sfx bell) to signal "speak now" moments
  • Keep the pause but with a visual cue like dimming instead of flashing white
Recommendation 5: Remove "Not Driving" Disclaimer

Lines 11: The "not driving / not operating machinery" block. This is a screen-based experience — already obvious. Remove it. (Keep it for future audio-only version.)

Staggering Bug Investigation

Potential Cause: Orphan Timer After Snap

In rsvp.js:scheduleNext(), when a snap fires via showWord(), the snap handler calls pause() which clears the existing timer. But scheduleNext() continues executing after showWord() returns, incrementing currentIndex and setting a new timer. This orphan timer fires during the snap pause and is normally harmless (isPlaying is false). However, if any edge case causes isPlaying to become true before the orphan fires, two concurrent scheduling chains would run simultaneously, causing words to appear out of order.

The fix: after showWord(), check if playback was paused by a snap before continuing:

// In scheduleNext():
this.showWord(this.currentIndex);
if (!this.isPlaying) return;  // snap paused us — bail out
this.currentIndex++;
this.timer = setTimeout(() => { this.scheduleNext(); }, interval);

Status: This is a defensive fix. The current code probably works in most cases because the orphan timer fires while isPlaying is still false. But it's a latent race condition that could explain intermittent staggering, especially if browser setTimeout scheduling gets delayed.
